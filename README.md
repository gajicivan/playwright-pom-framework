# OTRS E2E — Playwright + TypeScript (Page Object Model)

End-to-end automation framework written as a portfolio piece, structured to mirror how I would cover an internal business application like OTRS.

The framework runs against [OrangeHRM's public demo](https://opensource-demo.orangehrmlive.com/) — a real-world HR system chosen because it has the same feel as an OTRS-style app: login, multiple modules, side navigation, forms, search, paginated tables.

## Stack

- **Playwright** with TypeScript
- **Page Object Model** with reusable component objects (TopBar, SideMenu)
- **Test fixtures** that inject page objects into specs
- **Data builders** for unique, isolated test data
- **Storage state** auth — login runs once per browser, reused by every authenticated spec
- **GitHub Actions** CI with cross-browser matrix (Chromium / Firefox / WebKit)

## Setup

Requires Node 20+.

```bash
npm ci
npx playwright install --with-deps
cp .env.example .env   # adjust if you point to a different environment
```

## Running tests

```bash
npm test                    # all tests, all browsers
npm run test:smoke          # @smoke only
npm run test:regression     # @regression only
npm run test:chromium       # single browser
npm run test:headed         # headed mode (watch the browser work)
npm run test:ui             # Playwright UI mode
npm run test:debug          # PWDEBUG=1 step-through
npm run codegen             # record selectors against the live demo
npm run report              # open last HTML report
```

Filter by tag, project, or grep:

```bash
npx playwright test --grep @smoke --project=chromium
npx playwright test tests/regression/admin-users.spec.ts
```

## Design decisions

**Storage-state auth.** The `auth.setup.ts` project logs in once per browser and saves the cookies/localStorage. Every other spec inherits that state, so we don't pay the login cost on every test. The login spec itself overrides this with an empty storage state — it's the only place we exercise the real login flow.

**Component objects, not deep page hierarchies.** `TopBar` and `SideMenu` appear on every authenticated page, so they live as standalone component objects that page objects compose. Avoids brittle inheritance chains.

**Label-scoped field locators.** OrangeHRM's DOM is generated by a UI framework with no stable test IDs. `BasePage.fieldByLabel('Username')` finds the input within the `.oxd-input-group` whose label matches — much more robust than CSS-only paths.

**Data builders return unique test data per call.** Every `aNewUser()` and `aNewEmployee()` includes a unique suffix, so parallel workers and reruns don't collide. The full lifecycle test creates and deletes its own user, so the demo data stays clean.

**Tags drive suite selection.** `@smoke` for critical path (~1 min), `@regression` for deeper flows. Same code, two execution profiles via `--grep`.

**Cross-browser via projects.** Each authenticated browser project depends on `setup`, so the matrix is `setup × {chromium, firefox, webkit}` with the dependency graph handled by Playwright.

## CI

`.github/workflows/playwright.yml`:

- Triggers: push to `main`, PRs, daily 06:00 UTC smoke, manual dispatch with suite selector
- Cross-browser matrix (chromium / firefox / webkit) with `fail-fast: false`
- Caches Playwright browsers between runs
- Uploads HTML report as an artifact (always), traces + videos (on failure)
- Reads credentials from repo `vars` / `secrets`

## Notes on the test target

OrangeHRM's demo resets every 24 hours, so any data created during a regression run disappears the next day — perfect for automated tests. The full-lifecycle test still cleans up after itself so reruns on the same day stay green.
